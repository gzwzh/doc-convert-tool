# 代码解耦设计规范

## 设计原则

### 核心原则
1. **单一职责原则（SRP）**: 每个模块/类/函数只负责一个功能
2. **依赖倒置原则（DIP）**: 高层模块不依赖低层模块，都依赖抽象
3. **开闭原则（OCP）**: 对扩展开放，对修改关闭
4. **接口隔离原则（ISP）**: 使用多个专门接口，不使用单一总接口

### 分层架构
```
表现层（UI）
    ↓
业务逻辑层（Service）
    ↓
数据访问层（Converter/Repository）
    ↓
工具层（Utils）
```

### 1. 基础抽象类设计

**目的**: 定义统一接口，确保所有实现类遵循相同规范

**规范**:
- 使用 `ABC` (Abstract Base Class) 定义抽象类
- 定义必须实现的抽象方法
- 提供通用的辅助方法
- 统一错误处理模式

**示例代码**:
```python

        """
        执行转换（子类必须实现）
        
        Args:
            input_path: 输入文件路径
            output_path: 输出文件路径
            **options: 转换选项
            
        Returns:
            转换结果字典
        """
        pass
    
    def validate_input(self, input_path: str) -> bool:
        """验证输入文件（通用方法）"""
        # 通用验证逻辑
        pass
    
    def cleanup_on_error(self, output_path: str):
        """错误清理（通用方法）"""
        # 通用清理逻辑
        pass
```

**使用场景**:
- 图片格式转换器
- 文档格式转换器
- 音视频格式转换器

### 2. 独立转换器实现

**目的**: 每个转换功能独立文件，便于维护和扩展

**规范**:
- 一个文件对应一个转换功能
- 继承基础抽象类
- 只包含该功能相关的逻辑
- 文件命名：`source_to_target.py`（如 `png_to_jpg.py`）

**目录结构**:
```
converters/
├── __init__.py
├── base.py                 # 基础抽象类
├── png_to_jpg.py           # PNG→JPG
├── png_to_webp.py          # PNG→WEBP
├── png_to_bmp.py           # PNG→BMP
└── png_to_gif.py           # PNG→GIF
```

**示例代码**:
```python
# converters/png_to_jpg.py
from PIL import Image
from .base import BaseConverter
from typing import Dict, Any

class PNGToJPGConverter(BaseConverter):
    """PNG到JPG转换器"""
    
    def __init__(self):
        super().__init__()
        self.supported_formats = ['jpg', 'jpeg']
    
    def convert(self, input_path: str, output_path: str, **options) -> Dict[str, Any]:
        """将PNG转换为JPG"""
        try:
            self.validate_input(input_path)
            
            quality = options.get('quality', 85)
            optimize = options.get('optimize', True)
            
            with Image.open(input_path) as img:
                # 处理透明通道
                if img.mode in ('RGBA', 'LA', 'P'):
                    background = Image.new('RGB', img.size, (255, 255, 255))
                    if img.mode == 'RGBA':
                        background.paste(img, mask=img.split()[-1])
                    else:
                        background.paste(img)
                    img = background
                elif img.mode != 'RGB':
                    img = img.convert('RGB')
                
                img.save(output_path, 'JPEG', quality=quality, optimize=optimize)
            
            return {
                'success': True,
                'output_path': output_path,
                'size': self.get_output_size(output_path)
            }
            
        except Exception as e:
            self.cleanup_on_error(output_path)
            raise Exception(f"PNG to JPG conversion failed: {str(e)}")
```

**优点**:
- 新增格式只需新增文件，不影响已有代码
- 每个转换器独立测试
- 代码职责清晰

### 3. 服务编排层设计

**目的**: 统一管理转换器，提供统一调用接口

**规范**:
- 注册所有转换器
- 路由到具体转换器
- 统一返回格式
- 处理通用业务逻辑（如文件命名、路径处理）

**示例代码**:
```python
# services/converter_service.py
from typing import Dict, Any
import os
import uuid
from pathlib import Path
from converters.png_to_jpg import PNGToJPGConverter
from converters.png_to_webp import PNGToWEBPConverter
from converters.png_to_bmp import PNGToBMPConverter
from converters.png_to_gif import PNGToGIFConverter
from config import DOWNLOAD_DIR

class ConverterService:
    """转换服务编排层"""
    
    def __init__(self):
        # 转换器注册表
        self.converters = {
            'jpg': PNGToJPGConverter(),
            'jpeg': PNGToJPGConverter(),
            'webp': PNGToWEBPConverter(),
            'bmp': PNGToBMPConverter(),
            'gif': PNGToGIFConverter()
        }
    
    def convert_png(self, input_path: str, target_format: str, **options) -> Dict[str, Any]:
        """
        转换PNG文件（统一入口）
        
        Args:
            input_path: 输入PNG文件路径
            target_format: 目标格式
            **options: 转换选项
            
        Returns:
            转换结果字典
        """
        target_format = target_format.lower()
        
        if target_format not in self.converters:
            raise ValueError(f"Unsupported target format: {target_format}")
        
        # 生成唯一的输出文件名
        unique_id = str(uuid.uuid4())
        output_filename = f"{unique_id}.{target_format}"
        output_path = os.path.join(DOWNLOAD_DIR, output_filename)
        
        # 路由到具体转换器
        converter = self.converters[target_format]
        result = converter.convert(input_path, output_path, **options)
        
        # 统一返回格式
        return {
            'success': True,
            'file_url': f"/downloads/{output_filename}",
            'filename': output_filename,
            'size': result['size']
        }
    
    def get_supported_formats(self) -> list:
        """获取支持的格式列表"""
        return list(self.converters.keys())
```

**优点**:
- 添加新格式只需在注册表中添加
- 业务逻辑统一管理
- 便于单元测试

### 4. 工具层分离

**目的**: 通用工具函数独立，避免业务逻辑混杂

**规范**:
- 按功能分文件（如 `file_handler.py`, `validator.py`）
- 使用静态方法或函数
- 无状态设计
- 可在多处复用

**目录结构**:
```
utils/
├── __init__.py
├── file_handler.py         # 文件操作工具
├── validator.py            # 参数验证工具
└── logger.py               # 日志工具
```

**示例代码**:
```python
# utils/validator.py
from typing import Optional

class Validator:
    """参数验证器"""
    
    @staticmethod
    def validate_quality(quality: Optional[int]) -> int:
        """验证图片质量参数"""
        if quality is None:
            return 85
        
        if not isinstance(quality, int):
            raise ValueError("Quality must be an integer")
        
        if quality < 1 or quality > 100:
            raise ValueError("Quality must be between 1 and 100")
        
        return quality
    
    @staticmethod
    def validate_target_format(target_format: str, allowed_formats: list) -> str:
        """验证目标格式"""
        if not target_format:
            raise ValueError("Target format is required")
        
        target_format = target_format.lower().strip()
        
        if target_format not in allowed_formats:
            raise ValueError(f"Unsupported format. Allowed: {allowed_formats}")
        
        return target_format
```

### 5. API路由层设计

**目的**: 处理HTTP请求，调用服务层

**规范**:
- 只处理请求/响应
- 参数验证后调用服务层
- 统一错误处理
- 不包含业务逻辑

**示例代码**:
```python
# api/routes.py
from fastapi import APIRouter, UploadFile, File, Form, HTTPException
from typing import Optional
from services.converter_service import ConverterService
from utils.file_handler import FileHandler
from utils.validator import Validator

router = APIRouter()
converter_service = ConverterService()
file_handler = FileHandler()
validator = Validator()

@router.post("/convert/png")
async def convert_png(
    file: UploadFile = File(...),
    target_format: str = Form(...),
    quality: Optional[int] = Form(None),
    optimize: Optional[bool] = Form(None)
):
    """转换PNG文件（只处理HTTP层）"""
    input_path = None
    
    try:
        # 参数验证
        target_format = validator.validate_target_format(
            target_format,
            converter_service.get_supported_formats()
        )
        quality = validator.validate_quality(quality)
        
        # 保存上传文件
        input_path = await file_handler.save_upload_file(file)
        
        # 调用服务层
        options = {'quality': quality, 'optimize': optimize}
        result = converter_service.convert_png(input_path, target_format, **options)
        
        return result
        
    except ValueError as e:
        raise HTTPException(status_code=400, detail=str(e))
    except Exception as e:
        raise HTTPException(status_code=500, detail=f"Conversion failed: {str(e)}")
    finally:
        # 清理临时文件
        if input_path:
            file_handler.cleanup_file(input_path)
```

## 前端解耦规范（React）

**目的**: 组件单一职责，高内聚低耦合

**规范**:
- 一个组件只做一件事
- 组件间通过props通信
- 状态提升到合适的层级
- 文件命名：大驼峰（如 `FileUpload.jsx`）

### 2. 服务层封装

**目的**: API调用统一管理，与组件解耦

**规范**:
- 所有API调用封装在服务层
- 统一错误处理
- 返回Promise
- 文件名：`api.js` 或 `xxxService.js`

**目录结构**:
```
services/
├── api.js                  # API调用封装
├── authService.js          # 认证服务（示例）
└── storageService.js       # 存储服务（示例）
```

**示例代码**:
```javascript
// services/api.js
const API_BASE_URL = 'http://127.0.0.1:8000';

/**
 * 转换PNG文件
 * @param {File} file - PNG文件
 * @param {string} targetFormat - 目标格式
 * @param {Object} options - 转换选项
 * @returns {Promise<Object>} 转换结果
 */
export const convertPNG = async (file, targetFormat, options = {}) => {
  try {
    const formData = new FormData();
    formData.append('file', file);
    formData.append('target_format', targetFormat);
    
    if (options.quality !== undefined) {
      formData.append('quality', options.quality);
    }

    const response = await fetch(`${API_BASE_URL}/api/convert/png`, {
      method: 'POST',
      body: formData,
    });

    if (!response.ok) {
      const errorData = await response.json();
      throw new Error(errorData.detail || '转换失败');
    }

    return await response.json();
  } catch (error) {
    console.error('Conversion error:', error);
    throw error;
  }
};

/**
 * 健康检查
 */
export const healthCheck = async () => {
  try {
    const response = await fetch(`${API_BASE_URL}/api/health`);
    return await response.json();
  } catch (error) {
    console.error('Health check error:', error);
    throw error;
  }
};
```

**组件中使用**:

**优点**:

### 3. 状态管理规范

**目的**: 状态提升到合适层级，避免prop drilling

**规范**:
- 本地状态放在组件内（如表单输入）
- 共享状态提升到父组件
- 复杂状态使用Context或状态管理库
- 避免过度提升

**示例**:
```jsx
// ❌ 错误：状态过度分散
function BadExample() {
  return (
    <div>
      <ComponentA />  {/* 内部管理state */}
      <ComponentB />  {/* 内部管理state */}
      {/* 无法共享状态 */}
    </div>
  );
}

// ✅ 正确：状态合理提升
function GoodExample() {
  const [sharedState, setSharedState] = useState(null);
  
  return (
    <div>
      <ComponentA state={sharedState} onChange={setSharedState} />
      <ComponentB state={sharedState} />
    </div>
  );
}
```

## 常见问题

### Q1: 什么时候需要创建新的服务层？
**A**: 当功能逻辑复杂，需要编排多个转换器或处理复杂业务流程时。

### Q2: 组件拆分到什么粒度？
**A**: 遵循"单一职责"原则，当组件超过200行或做多件事时考虑拆分。

### Q3: 工具函数应该放在哪？
**A**: 
- 通用工具（文件、验证）→ `utils/`
- 业务相关工具（特定转换辅助）→ `converters/helpers.py`

### Q4: 如何避免循环依赖？
**A**: 遵循分层架构，依赖关系单向流动：
```
API层 → 服务层 → 转换器层 → 工具层
（高层依赖低层，低层不依赖高层）
```

## 总结

### 核心要点
1. **单一职责**: 一个文件/类/组件只做一件事
2. **抽象分离**: 通过抽象类定义接口，实现类处理细节
3. **分层清晰**: 表现层、业务层、数据层职责分明
4. **工具独立**: 通用工具函数独立复用
5. **易于扩展**: 新增功能只需添加文件，不修改已有代码

### 收益
- ✅ 代码可维护性提高
- ✅ 测试更容易
- ✅ 团队协作更顺畅
- ✅ 新功能开发更快
- ✅ Bug定位更准确
